Java Measurement Dataflow Interface (JMDI) Overview

The Entazza Crew

Portions of this document are obsolete and is currently being revised.

Introduction

The Measurement Dataflow API specifies a dataflow environment where measurement data flows through a network of processing blocks. The measurement data is processed and transformed as it passes through the processing blocks.

 

The Java Measurement Dataflow API is described by the following models:

 

q       Component Model – Describes rules for how functionality is aggregated into entities.

q       Data Model – Describes the underlying data structures used in the framework,.

q       Communication Model – Describes how the components communicate.

q       Threads Model – Describes the rules for the permitted threads of control throughout the system.

q       Configuration Model – Describes how the components are configured.

q       Lifecycle Model – Describes how the components are instantiated and destroyed.

 

The following sections describe each model in greater detail.

Component Model

The component model describes how the program functionality is organized.

 

The specification for the Measurement Dataflow API component model is based on the IEEE 1451.1 standard. The basic building block used to build a IEEE 1451.1 system is known as a Block.  There are three types of Blocks: Function Block, Transducer Block, and NCAP Block. The first two block types are also known as F-Block and T-Block, respectively.

 

Collections of Blocks are connected, as described in the following section, to form a processing network or graph. Each Block is responsible for a specific function within the overall graph. Data flows through the network from one Block to the next. This style of program organization is sometimes referred to as dataflow programming. Figure 9 is an example of such a system. It is an instance diagram of a deployed application built with this paradigm.

 

 

 

Figure 1. An example of a IEEE 1451.1 Application

 

Figure 10 shows the Block class hierarchy. The most common type of Block in an IEEE 1451.1 system is the F-Block. Application logic resides within F-Blocks. An example of an F-Block may be an F-Block, which performs averaging, or perhaps one which performs decimation on the data stream passing through it.

T-Block encapsulates the interface to the physical transducer while providing a block interface to other blocks.[1] It is similar to F-Block in that it also contains program logic but provides data that come from transducers rather than from other Blocks. The T-Block also has the responsibility for sampling the transducers.

The NCAP Block plays a system manager role in the system. There is only one instance of the NCAP Block per node. It provides container functionality such as lookup services.

 

In the 1451.1 to Java mapping specified by JDDAC, all blocks support a perform() method to allow invocation from anyone. Details are described in the section on Communication Models.

 

 

Figure 2. Blocks.

 

Blocks locate other Blocks using the NCAP Block.  The NCAP Block provides a registry service where block instances are registered. References to the blocks can be obtained by name or by attribute.

Data Model

The data model describes the fundamental data structures used in the Dataflow API.

In IEEE 1451.1, there is only one universal data element, and that atomic data element is called the Argument. An array of Arguments, addressable by name, is called an Argument Array (or ArgArray for short).  Many different kinds of objects can be an Argument. Simpler Arguments are the Object versions of fundamental data types such as Integer or Float. Arguments can also be more complex data types such as Unit, Uncertainty, or Time Representation.  Argument Arrays are also considered to be Arguments. Thus, there is a recursive containment relationship between Argument and Argument Array.

Most methods of the Block classes either take as parameters individual objects, or a single Argument Array.

In JDDAC, an Argument is mapped into java.lang.Object. This mapping  results from the fact that the object versions of most, if not all, fundaamental data types are declared final in Java, meaning that they cannot be changed. All these classes has java.lang.Object has a root ancestor in the class hierarchy. In IEEE 1451.1, the Argument class is the root of these fundamental types, thus it makes sense to map Argument to java.lang.Object, even if java.lang.Object is wider in scope than the original Argument definition. We also extend the notion of Argument Array to allow access by name as well as by index number as specified in the 1451.1 specification..

Communication Model

The communication model describe how the Entities communicate with one another and pass data to other instances.

In IEEE 1451.1, Entities do not address one another directly. Rather, they communicate via helper objects called Ports. There are different types of ports for different types of communications. Having communication go through an intermediate proxy, such as a Port, allows the Entities to be unaware whether the communication is local or remote. More about Ports later.

IEEE 1451.1 specifies two major kinds of communications among Entities: client-server and publish-subscribe. Client-server communication is similar to a Java method invocation. There is one client which requires service, and one server which provides this service.  The identity of the server instance is known to the client object. The client sends a request to the server with an operation ID which identifies the service being requested along with parameters for the request. The server then returns the results of the operations, or an exception if the request fails.

In publish-subscribe communication, publishers generate data and subscribers consume data.  In this communication model, there is a notion of a topic. Subscribers register interest in topics. A publisher instances generates data on a topic, and subscribers which have expressed interest in the topic is given a copy of the data. There may be multiple publisher and subscriber instances, none of which need to be aware of the identify of any of the other instances. 

 

There are two types of communication scenarios to consider: an intra-VM case and an inter-VM case.

Intra-VM communication takes place within a virtual machine, while inter-VM communication involves communication between two components that reside in different virtual machines, with the communication possibly going across a network. In JDDAC, the Entities are not aware whether the communication is local or remote.

 

 

Figure 4. Remote Communication Port API

 

In the intra-VM case, a client-server calls is simply regular method invocation, without needing to go through an intermediary. Objects being passed among Entities are passed by object references. This is also consistent with the default behavior in Java.

 

All Entities support a perform() method, all with the same signature, so there is a generic way for invoking any method on any Entity.

 

The IEEE 1451.1 network communication abstractions are called Ports.  Ports are responsible for handling the details of network communications. There are two types of ports: Publisher Port and Subscriber Port. These two objects allow publish-subscribe communication.

 

To actually communicate across the network, we use a variety of Java communication mechanisms such as RMI or JXTA. The block developer is protected from needing to know the details of the communication mechanism through the Port abstraction.

 

Thread Model

 

The thread model describes how the threads traverse the network of blocks. JDDAC allows multiple threads to be active within the dataflow network, although in general, thread swapping is minimized to increase the performance.

 

Since the configuration model describes how the components, as well as the topology of the network organizes Block instances in a graph, the implementer of the blocks need to be cognizant of certain design patters which lead to deadlock.

 

Deadlock can occur when communication flows from block to block in a unique pattern that varies from thread to thread.  For example, consider the diagram below that shows a message flowing from A to B to C and another flowing from C to B to D.  If each block synchronizes to itself while processing the message then a deadlock will occur if A and B are locked by thread X while thread Y has locked C but has not yet locked B.

Figure 5. Remote Communication Port API

 

To address this problem blocks are required to treat any subscriptions they receive as immutable objects and to relinquish any internal synchronization locks before initiating a publish operation.  As a result, blocks are free to keep references to subscriptions they receive and can even include all or portions of subscriptions in publications they generate. 

 

However, any new state placed into a publication must subsequently be considered immutable.  This usually means that new elements placed into a publication need to be copies of internal state rather than references to internal state.  One approach is to simply do a deep copy on the entire message and then make changes.   This however is inefficient if the message is large and only a small portion is being changed.  This can optionally be addressed by only making copies of  container objects above the new values.

 

Figure 6. Remote Communication Port API

 

As an example, consider the figure above representing a container relationship between objects.  If object D is replaced in the original message then container objects A and B must be made into copies before the message can be published from the Block.  Example pseudo code showing these concepts is provided below.  The key aspect to note is that the publication list must be cloned (and any necessary cloning in the message) before they synchronized region is exited.

 

publish(message) {

           // clone the subscriber list and then publish the messages

           synchronize (this) {

                        tmpSubList = subList.clone();

            }

 

            // No locks held at this point

            foreach sub in tmpSubList {

                        port.send(message);

}

 

someMethod() {

            synchronize (this) {

                        notify(message);  // This is illegal: publishing while locked

            }

}

 

// example notify callback on a new message being received

notify(message) {

            if (Thread.holdsLock(this)) throw new Exception();

            if (need to modify message) {

                synchronize (this) {

                        tmpMsg = message.deepclone();

                        modify(tmpMsg);  // Modify the copy

                }

            }

 

            // No locks held at this point

            publish(tmpMsg);

}

Configuration Model

The configuration model describes how the components, as well as the topology of the network which organizes Block instances in a graph, can be specified at start-up time by the system developer.

A configuration for a Block is a collection of name-value pairs, or Arguments. The collection is held in an ArgArray. The entire system configuration then consists of  the sum of these ArgArrays for all the Blocks in this system.

More details on configuration can be found in here.

Lifecycle Model

The lifecycle model describes how the Blocks come to exist and cease to exist. It also describes the different states that the blocks go through during the time that they exist.

In terms of object creation, NCAP Block is created at system startup. The NCAP Block is considered part of the runtime environment. NCAP Block then serves as the factory for creating instances of Block objects.

IEEE 1451.1 also defines states within a Block’s lifetime. This allows the system manager to place the Blocks into inactive states while destroying the instances.

Blocks are in one of the three following states during the time they are in existence:

q       UNINITIALIZED –  When in this state, most Block operations are not functional. See Section 9.1.3.2.1 of the IEEE 1451.1 specifications for additional description.

q       INACTIVE - When in this state, the Block is ready for configuration, initialization, and diagnostic activities. See Section 9.1.3.2.2 of the IEEE 1451.1 specifications for additional description.

q       ACTIVE – When in this state, the Block is in its regular operational mode.

In addition, Function Blocks have its own set of states when the Function Block is in the Active state:

q       STOPPED -  When in this state, the F-Block cannot change the state of any object external to the Block.

q       RUNNING – When in this state, the F-Block is fully functional.

q       IDLE – When in this state, most functionalities of the F-Block are inoperative. See Section 9.3.2.1.3 of the IEEE 1451.1 specifications for details on the restrictions.
