Java Transducer Interface (JTI) Overview

The Entazza Crew

 

A transducer is an entity that interacts with the physical world. It is a general term that includes both sensors and actuators. A transducer may source (generate) or sink (receive) different types of data. For example, a GPS receiver, modeled as a sensor, generates both location and time data. A thermostat transducer might receive temperature data as part of its settings as well as measuring temperature so it acts as both a sensor and an actuator.

 

In JDDAC, a datum originates from, or is destined for, a channel. Channels are the actual data sources and sinks within a transducer. There may be multiple channels within each transducer, with some being actuators and others being sensors. There may be multiple transducers on each measurement device.

 

The purpose of the Java Transducer API is to provide a facility for application developers to interface with various different kinds of sensors in a uniform fashion.  There are several impediments that make this goal a challenge. Different transducers have different protocols, or ways of interacting with them. Some transducers expect to be polled. Others generate interrupts when real-world events occur. Different transducers also have different behavior. Some transducers can make multiple measurements with a single trigger, such as ones that generate a time-series measurement, while others need to be triggered for each reading. Lastly, different transducers generate data in different formats with different context. One temperature sensor may provide temperature in Celsius, yet another may provide the same measurement in Kelvin, along with a timestamp.

 

In JDDAC, there is a Transducer Electronic Data Sheet (TEDS) associated with each transducer.  The TEDS characterizes the transducer and the data that it can generate or accept. It describes the communication protocol with the transducer, the behavior of the transducer, the data that the transducer generates, and the context information, called metadata, associated with the generated data. The TEDS concepts, and many of the fields within the TEDS, are drawn from the IEEE 1451.2 standard. Some of the fields in the IEEE 1451.2 TEDS are specific to the actual IEEE 1451.2 hardware and thus are not in JDDAC.

 

Figure 3 is a UML diagram of the major classes and interfaces in the Java Transducer Interface. The primary interface is called TIMInterface that represents a physical transducer. TIM stands for Transducer Interface Module and is a term proposed to the IEEE1451.0 working group.  There may be multiple realizations of the Transducer interface in a system, with one instance for each physical transducer. This basic interface supports a canonical read/write data interface. Both read and write operations can act on a collection of channels rather than a single one.

 

From this interface, it is possible to retrieve the metadata associated with the transducer. The metadata is a collection of name-value pairs that provide context for the transducer as well as the measurements that the transducers produce.

 

It is also possible to register a callback to be invoked when data is available from this transducer. This is useful for making measurements whose triggering conditions are not under the control of the measurement device.

 

Beyond the basic TIMInterface, there is an additional interface called the TriggeredTIMrInterface. This interface introduces the notion of time into the method signatures. A measurement or actuation request can be scheduled to occur at a specific instant in the future. The measurement values can also have timestamp information associated with them.

The abstract class BasicTIM provides some default behavior, specified by these interfaces, which would be common for most transducer devices. Device developers realize either the TIMInterface to create a class or derive from BasicTIM, called MemoryUsage in Figure 1, which provides the implementation to interact with the transducer. Its main responsibility is to encode and decode commands to and from the particular transducer device.

 

The Java Transducer API will also provide various helper objects to interface with different physical interfaces. These helper classes will support a query-response style of interaction.

 

There is metadata associated with each transducer to provide information such as units and uncertainty.  A TEDS provide Metadata. TEDS can be manifested in file format or provided programmatically.

 

Not shown in this figure is the Transducer Block discussed in the Dataflow API section. The TransducerBlock interacts with the classes shown in this figure to source or sink measurement data. Users of the Measurement Dataflow API interact with Transducer Blocks rather than this interface. Details are provided in the Dataflow section.
