JDDAC Tutorial – Configuration Guide

Jerry Liu

Introduction

This tutorial discusses how JDDAC applications are configured through startup XML files. Users wishing to customize their applications can edit the XML file.

For the JddacClient application, the name of the XML file is provided as the first command line argument:  java –jar lib/JddacClient.jar  myConfig.xml

Steps during Application Launch

There are 3 steps during the launch of a JDDAC application. Although this tutorial discusses configuration as a “one-time” process, in reality JDDAC supports dynamic configuration as well.

Step 1: Making the classes available to the Java class loader.

All classes used in the JDDAC application must be specified on the CLASSPATH or in the –jar parameter to the java command. For the tutorial, everything is packaged into the JddacClient.jar file.

Step 2: Instantiating instances of classes

In the configuration XML, each object to be instantiated is specified with the two attributes: the class name, and the desired instance name. The configuration logic uses the Class.forName( ) mechanism[1]. This will work for any class in the system. Note that JDDAC will register the instance by the “instanceName” and provides a simple local registration service so that other objects can get a local reference to the object.

  <arg n="instanceName">My Memory Usage Stim</arg>

  <arg n="className">net.java.jddac.jti.transducer.MemoryUsage</arg>

Step 3: Configure select instance variables.

JDDAC object also can accept numerous configuration parameters. This is supported on any class that implements the “ConfigurationInterface”. Here’s an example for the creation of the “Reporter” F-Block which manages communication between the JddacClient and the JDDAC server. Key configuration parameters will be discussed below.

<argArray l="8">

  <arg n="className">net.java.jddac.jmdi.fblock.Reporter</arg>

  <arg n="instanceName">Reporter</arg>

  <arg t="int" n="Block.numSubs">1</arg>

  <arg n="Block.s1.qual">storeMeasurement</arg>

 <arg n="Reporter.destination">http://jddac.labs.agilent.com:8080/server/probe</arg>

  <arg t="long" n="Reporter.connectInterval">10</arg>

  <arg n="Reporter.transportName">reporter</arg>

  <arg n="Reporter.xdestination">http://localhost:8080/server/probe</arg>

 </argArray>

 

Configuration XML File

The JDDAC XML configuration file contains a “configuration set”. This is represented as an “argument array” of type “vector”. In the following XML example, this vector contains 5 elements.

The startup processing parses this XML and traverses the vector in sequential order. The first “configuration” encountered is fully processed before moving onto the next “configuration”. This means that object at the top of the XML are instantiated and configured BEFORE object further down in the XML. During instantiation, an appropriate constructor will be called with NO arguments. The programmer must initialize the object’s attributes to reasonable defaults at this time.

Each element of the “configuration set” contains a “configuration”. Each is represented as an “argument array” and contains all the “name / value” configuration parameters for this object. The order of parameters in the configuration is arbitrary. For objects that implement the “ConfigurationInterface”, the configure( ) method will be called with the configuration provided as a parameter. The programmer is responsible for retrieving appropriate values from the configuration during this call.

To prevent name-space collisions within the object’s class inheritance chain, names are prefaced with the leaf-node class name.

NCAP Block

The NCAP block is a singleton object that is required for each JDDAC application. The “NCAPBlock.probeNetworkID” parameter contains your registration key information.

Reporter F-Block

The Reporter is also singleton object in the current JDDAC release. It is a sub-class of “Function Block” (F-Block for short) and represents application level processing in the 1451.1 architecture. It manages communication to the JDDAC server. The “Reporter.destination” parameter specifies the connection information. The “Reporter.connectInterval” parameter specifies the period for connections in seconds. Note that the Reporter subscribes to intra-JVM communications from other local JDDAC object. This is controlled by the “Block.numSubs” and “Block.s1.qual” parameters. See below for more details on intra-JVM publication / subscription configuration. Note that the Reporter can be configured to subscribe to more than one topic by changing these parameters.

Memory Usage Transducer

This object is a transducer that makes memory measurements on the local JVM. It takes no configuration parameters. It is important to realize that Transducers ARE NOT part of the 1451.1 Block architecture. They provide an interface to the transducers but do not support Block-to-Block communications. See the TransducerBlock for that functionality.

BasicTransducerBlock Block

TransducerBlocks (T-Block for short) are at the boundary between the 1451.1 world and Transducers. The “BasicTransducerBlock.transducerName” configures the T-Block with the appropriate transducer. The “period” parameter specifies the periodic sample rate in msecs. The T-Block generates a publication with each measurement. This is configured with the “Block.numPubs” and “Block.p1.topic” parameters.

Echo  F-Block

This F-Block is for diagnostic purposes. It subscribes to appropriate topics using the “Block.numSubs” and  “Block.s1.qual” parameters. Incoming subscriptions are printed to the console. Note that the Echo block supports multiple subscriptions by manipulating these parameters.

 

Managing Intra-JVM Communications

The current JDDAC release has implemented the “publish / subscribe” communication model in the IEEE 1451.1 standard. Future releases will add the “client / server” and “asynchronous client / server” models.

 

The most common usage of “pub/sub” is to route measurement messages between cooperating F-Blocks within the JddacClient. This is known as intra-JVM pub/sub. Any sub-class of Block inherits support for this form of communication. Just specify the desired number of publications with the “Block.numPubs” parameter followed by each topic using this form: “Block.p1.topic”,  “Block.p2.topic”, “Block.p3.topic”, etc.

 

Topic strings should NOT contain the “.” or “*” characters. The period is used as a field separator and the “*” is used for specifying wildcards.

 

For subscriptions, use the “Block.numSubs” parameter and the appropriate number of “Block.s1.qual”, “Block.s2.qual”, etc. parameters. Subscription qualifiers can use the “.” and “*” for limited wild-card support.

 

For example, a “Block.s1.qual” of “*.temperature” would match any of these topics: “outlet.temperature”, “inlet.temperature”, etc.

 

Building your own F-Blocks and Transducers

To enhance the JddacClient, programmers will be implementing new F-Blocks and Transducers. It is recommended that you study the examples to understand startup and threading conventions.
